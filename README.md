## Схема управления ограничениями доступа

В проекте реализована собственная RBAC-система разграничения доступа.  
Она построена на связке из пользователей, ролей, прав и ресурсов.

### Основные сущности

#### users
Таблица `users` хранит данные учетной записи:
- имя;
- фамилия;
- отчество;
- email;
- хеш пароля;
- флаг активности `is_active`;
- признак первичного администратора `is_primary_admin`.

Именно пользователь является субъектом доступа.

#### roles
Таблица `roles` содержит роли системы:
- `admin`;
- `moderator`;
- `user`.

Роль описывает набор допустимых действий для пользователя.

#### permissions
Таблица `permissions` хранит права доступа, например:
- `posts_read_all`;
- `posts_create_own`;
- `posts_update_own`;
- `posts_update_all`;
- `posts_delete_own`;
- `posts_delete_all`;
- `users_read`;
- `users_write`;
- `users_delete`;
- `roles_manage`.

Право определяет, какие действия разрешены над ресурсом.

#### resources
Таблица `resources` хранит защищаемые ресурсы:
- `posts`;
- `users`;
- `roles`.

Ресурс — это объект, доступ к которому нужно контролировать.

### Связующие таблицы

#### user_roles
Связывает пользователей и роли.  
Один пользователь может иметь несколько ролей.

#### role_permissions
Связывает роли и права.  
Одна роль может включать несколько прав.

#### permission_resources
Связывает права и ресурсы.  
Таблица показывает, к какому ресурсу относится право и какие действия оно разрешает:
- чтение;
- запись;
- удаление.

### Логика проверки доступа

При каждом защищённом запросе выполняется следующая последовательность:

1. Из запроса извлекается токен.
2. По токену определяется пользователь.
3. Если пользователь не найден, возвращается `401 Unauthorized`.
4. Если пользователь найден, проверяются его роли и права.
5. Если доступ к ресурсу отсутствует, возвращается `403 Forbidden`.
6. Если доступ разрешён, возвращается `200 OK`.

### Админское управление правилами

Пользователь с ролью администратора может управлять правилами доступа через отдельный API:
- просматривать роли;
- просматривать права;
- просматривать ресурсы;
- просматривать связи роль–право;
- назначать роли пользователям;
- удалять роли у пользователей;
- назначать права ролям;
- удалять права у ролей.

Перед выполнением любого изменения сервер сначала проверяет токен и право `roles_manage`.

### Принцип работы на практике

- Если токен отсутствует, запрос не проходит и возвращается `401`.
- Если токен есть, но права недостаточны, возвращается `403`.
- Если пользователь имеет нужную роль или право, ресурс выдаётся с кодом `200`.
